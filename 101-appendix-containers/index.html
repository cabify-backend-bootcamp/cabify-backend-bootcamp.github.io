Containers

Containers

We are going to take advantage of the container technology for this bootcamp, so we're going to explain briefly what they are and how to use them.

Modern operating systems allow to have multiple user-space instances, that look like real computers from the point of view of programs running in them. The difference between a program running in an ordinary operating system and a program running in a container is that the latter can only see a subset of the resources that are visible for the former, those assigned to the container.

For the bootcamp, we are mostly interested in the flexibility that containers offer. Since each container behaves as it's a different computer, we can simulate a complete distributed system locally, in our own computers.

Docker

Docker is one of the best known container system available and is the one we will use during the bootcamp. A docker system is composed of a daemon (dockerd) and a client (docker). The client communicates with the daemon that listens for commands. We will always interact through the client.

Docker manages several different entities, we are interested in just two:

  • A docker image, a template used to build docker containers.
  • A docker container, which uses a docker image to build the actual isolated environment and run the applications.
  • A docker registry, which is a (local or remote) repository for docker images.

The usual workflow while working with docker is:

  • Create an image or get one from a docker registry
  • Run a container based on the image
  • Stop the container

Image creation

To create an image first we have to define it. This is usually done in a Dockerfile file. Here we can see an example of a simple node application image:

FROM node:latest
ADD . /app
WORKDIR /app
RUN npm install
CMD ["node", "index.js"]

Now we can build the image with:

$ docker build -t myapp:latest .

myapp is the name we tag the image with. This will be used to run a container based on this image.

You can see the available images with:

$ docker images

Run a container

You can run a container either using your own image as explained in the previous section, or some image available in some docker registry. For example, the command to run a container based on the image built previously would be:

$ docker run myapp

To run a container publicly available in a docker registry (for example, a mongodb image):

$ docker run mongo

In this case, the docker command does two different things:

  • Download the image
  • Run the container built from this image

The first step can also be done separately with:

$ docker pull mongo

Docker compose

Docker provides an easy way to build, share and run containers, which is great. But we are building distributed systems, and having to run each container manually becomes tedious fast.

There are several ways to approach such problem, we'll go with docker compose. Docker compose is a tool to define an run multi-container applications and will allow us to build the systems we will use in the exercises locally quite easily.

To define such multi-container system, a configuration file, in YAML format, is used. The configuration file for the first exercise is as follows:

version: '3'
services:
  exercise1:
    build: .
    ports:
     - "9001:9001"
  messageapp:
    image: "cabify/backend-bootcamp-messageapp:latest"
    ports:
      - "3000:3000"

The configuration file is composed of:

  • A configuration format version in the first line
  • Containers definition in the rest of the file. There are two services:
    • exercise1: A container that is built using the image described in Dockerfile in our current directory and will listen in port 9000.
    • messageapp: A container that is built using the image tagged with cabify/backend-bootcamp-messageapp:latest (which may be available either locally or in a docker registry), which will listen in port 3000.

This is just the basics of how to configure a multiple container system with docker compose, but it should give a quick overview of the idea. Check the references for more information.

Now, we just need to build the system we have just described:

$ docker-compose build

and run it:

$ docker-compose run

This will launch both services in containers as defined above.
