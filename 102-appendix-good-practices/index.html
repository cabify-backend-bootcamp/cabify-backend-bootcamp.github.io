Good Practices

Good Practices

Software Engineering is more than just writing code[1]. By Software Engineer, we don't mean a person with a degree in Computer Science. But one who looks at writing quality software as their profession. Knowing how to program is not enough to make software engineering. There are several analogies that help us to understand this:

  • Everyone can sing (especially in the shower), but if you have to throw a party you probably hire professional musicians.
  • Everyone can cook. But when it’s time to feed a lot of people we hire a Chef.
  • Everyone can write, but it doesn't mean you are a novel author.

Software is not different. You might know how to program and you might be able to code simple programs, but that's different from writting engineered programs. And one thing that characterizes a engineered program is the code quality.

Code quality is subtle. There are some code constructions that could cause real troubles to the authors, and directly or indirectly also to the users. There are also some good practices that help to avoid such harmful constructions and improve the quality of our sofware. Along this appendix, we will describe some of them, divided in two different sections:

  • General good practices. Some general principles to write better sofware.
  • Design good practices. Good practices applied to the sofware design and software architecture.

Important disclaimer: this is about good practices. You should not interpret this as The Truth. It's a guide, a compendium of tricks and advices to make better software. Do not read it as a code of conduct. We expect you to apply your own criteria and think by yourself while coding.
