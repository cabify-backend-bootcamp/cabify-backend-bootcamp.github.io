Design Good Practices

Design Good Practices

At the beginning, every software project is a masterpiece. Everything is simple, clean, concise, elegant. But then the time passes, and the software start to rot. It's a progressive process: an ugly part here, a little hack there. And over time the rotting continues until our beautiful codebase becomes a mess that the developers find increasingly hard to maintain. Eventually, the effort required to make the simplest of changes to the software becomes so high that the engineers cry for a full rewrite (likely using microservices :-) ).

Symptoms of rotten software

But, how do we detect our codebase is rotting? There are four symptoms that could help to detect it.

  • Rigidity. The tendency of the software to be difficult to change. Every single change causes a cascade of changes in other parts of the code (dependencies). What seemed to be a two days change becomes several weeks of work. This derives in unhappy managers, who in degenerate cases reduce the changes to critical fixes only. The software stops adding new value to the business. And the product collapses.

  • Fragility. The tendency of the software to break upon changes. Very often, changes that have nothing to do with the part that brokes. Managers are afraid to make changes, so they limit them as they do before rigidity. Again, in degenerated cases we stop adding new value, and the product collapses.

  • Immobility. The inability to reuse software fragments. When one programmer detects he needs a module that is similar to one that other programmer wrote. However, the module to be reused is tangled with its dependencies, and the cost of separating things is too great. And the module is simply rewritten.

  • Viscosity. The tendency of the software to make it difficult to preserve the design, stimulating the programmer to make hacks instead. In other words, it's easy to do the wrong thing, but hard to do the right thing. This is something that applies not only to the code but also to the development environment: build systems, testing frameworks, service configuration, etc.

Causes of rotten software

There is one simple root cause for our beloved software to degrade: changing requirements. That's essentially why our codebase starts as a clean, elegant solution. Not only because it is small, but also because we start defining our architecture based on clear requirements. So we make a design that does the job really well. But, the sooner or later the requirements will change. And the software will have to give response to things it was not designed for.

We cannot expect the requirements not to change. This is a essential part of software development. What we have to do is to find a way to make our designs resilient to changes. But, what kind of changes could degrade our designs so much? Simple: those that introduce new and unplanned dependencies. Each of the four symptoms described above is either directly or indirectly caused by improper dependencies between the modules of our software. We have to manage the dependencies to prevent they propagate. We need dependency firewalls.

SOLID Principles

There are five well-known principles in Object Oriented Programming (but most of them applicable to other programming paradigms) that could be followed in order to manage module dependencies. We will analyze them in detail in the reminder of this appendix.

S: Single Responsibility Principle (SRP)

Each software module should have one and only one reason to change.

This is another formulation of the principles of cohesion and coupling: gather together the things that change for the same reasons. Separate those things that change for different reasons. That's exactly the meaning of one reason to change. If there are more than one business logic coded in a module, we will have more than one single person (or business unit) requesting changes in that module. As consequence, the changes requested by one person could impact the business logic under the responsibility of another one: introducing new bugs, changing the behavior unexpectedly, affecting the performance, etc.

Let's see an example. Consider a module that generates and prints a finantial report. If we think about it, we could distinguish two different reasons to change, as:

  • Change what are the contents of the report. We might want to add a new item that tells how much we invested in floppy disks in the last month.
  • Change how this report is printed. We might want to generate a PDF output instead of the HTML one we had so far.

The SRP says that these two aspects of the problem are really two separate responsibilities, and should therefore be in separate modules.

O: Open Closed Principle (OCP)

A module should be open for extension but closed for modification.

It means simply this: we should write our modules so that they can be extended, without requiring them to be modified. In other words, we want to be able to change what the modules do, without changing the source code of the modules.

An essential way to achieve this is creating abstractions. In Object Oriented Programming, we can define abstract classes or interfaces that define the contract of the module. The code that depends in such module will depend in these abstract classes or interfaces, and not in the implementations. If we need to extend the module, it's as simple as adding new classes that implements the abstract class or interface to the module. That requires no change in any existing code. Neither in our module, neither in others that depends on it.

If you are not familiar with terms like interace of abstract class, you can read this introduction to polymorphism and abstract types.

Let's see this with an example. Let's assume we have a web application that have to authenticate an HTTP request to ensure it contains a valid username and password. We put all the code that does the authentication in a function authorize(user, password) that, given a user and a password, indicates whether that user is authenticated. Internally, this funcion queries the DB to look for that user and check if the password matches.

This design is not following OCP. If we want to add a new way to authenticate users (e.g., LDAP, OAuth, etc), we will have to change most of the code in the authorize function. A nice way to follow OCP would be to define a AuthenticationProvider abstract class.

Our request handlers would have access to an instance of AuthenticationProvider, so they only have to invoke authorize and forget about the details of how it is implemented. Initially, we would provide a DbAuthenticationProvider class that implements the interface defined by AuthenticationProvider. If we want to add more authentication methods, we would add new classes to the design: LdapAuthenticationProvider, OAuthAuthenticationProvider, etc, without changing a single line of code in auth module.

L: Liskov Substitution Principle (LSP)

Subclasses should be substitutable for their base classes.

The concept is simple. Derived classes should be substitutable for their base classes. That is, a user of a base class should continue to function properly if a derivative of that base class is passed to it. This may seem obvious, but there are subtleties that need to be considered. The canonical example is the Circle/Ellipse dilemma.

Most of us learn, in high school math, that a circle is just a degenerate form of an ellipse. All circles are ellipses with coincident foci. So we can have a definition of ellipse as:

class Ellipse {
  constructor(focusA, focusB, radius) {
    this.focusA = focusA;
    this.focusB = focusB;
    this.radius = radius;
  }

  get focusA() { return this.focusA; }
  get focusB() { return this.focusB; }
  get radius() { return this.radius; }

  set focusA(val) { this.focusA = val; }
  set focusB(val) { this.focusB = val; }
  set radius(val) { this.radius = val; }

  area() {
    return ...;
  }
}

A ellipse is a pair of foci points and a radius. We provide some getters and setters, and a method to calculat its area from the properties. Now, we can define the circle as:

class Circle extends Ellipse {
  constructor(center, radius) {
    super(center, center, radius);
  }

  set focusA(val) {
    this.focusA = val;
    this.focusB = val;
  }
  set focusB(val) {
    this.focusA = val;
    this.focusB = val;
  }
}

What we do here is to request just the center of the circle and the radius in the constructor. In other words, the single focus point we will have. We pass it from the constructor as both focusA and focusB constructor arguments, so both foci are the same. We also override some methods to satisfy both foci are always the same.

Perfect! We did it. We have a circle definition that reuses the ellipse definition with subtyping. Now let's check whether it satisfies LSP:

void func(ellipse) {
  let a = Point(-1,0);
  let b = Point(1,0);
  ellipse.focusA = a
  ellipse.focusB = b
  assert(ellipse.focusA == a);
  assert(ellipse.focusB == b);
}

If we invoke func() with an instance of Ellipse class, the code works fine. We set the foci, and check they are still there. But if we invoke it with an instance of Circle, the code doesn't work. The call to the focusB setter also writes focusA property, so ellipse.focusA == a is not met anymore. Even though we defined a subtype relationship between Ellipse and Circle, the instances of the latter cannot be used where instances of the former are expected. We are not fully satisfying the contract. More precisely, the post-conditions of the setters are not met.

I: Interface Segregation Principle (ISP)

Many client specific interfaces are better than one general purpose interface.

The essence of the principle is quite simple. If you have a class that has several clients, rather than loading the class with all the methods that the clients need, create specific interfaces for each client and multiply inherit them into the class.

Let's say we have a class with many clients, and one large interface to serve them all. Note that whenever a change is made to one of the methods that one client calls, the rest of clients may be affected.

A better technique is to have the methods needed by each client placed in special interfaces that are specific to that client. Those interfaces are multiply inherited by the implementing class.

D: Dependency Inversion Principle (DIP)

Depend upon Abstractions. Do not depend upon concretions.

Dependency Inversion is the strategy of depending upon interfaces or abstract functions and classes, rather than upon concrete functions and classes. In fact, we already discussed about this and we saw an example in OCP.

We use to make software following a top-down design that exhibits a particular kind of dependency structure. High level modules depend upon lower level modules, which depend upon yet lower level modules, etc. For example, in a web application, the path router depends on the page controllers. The page controllers depend on services. The services depend on repositories. The repositories depend on DB clients. And so on. That's actually the direction of the function calls in runtime.

graph TD PathRouter --> IndexController PathRouter --> LoginController PathRouter --> ProductController IndexController --> other1[...] LoginController --> LoginService LoginController --> AnalyticsService ProductController --> other2[...] LoginService --> UsersRepository AnalyticsService --> other3[...] UsersRepository --> MySQLClient

Thanks to object oriented programming techniques, we can revert the direction of the dependencies by introducing abstract classes or interfaces in the design.

Thanks to this, we can see that some modules like LoginController depends exclusively on abstract classes LoginService and AnalyticsService. So any other dependency carried by the implementations is not propagated to LoginController (the dependency firewall we discussed above). In fact, what we did is to invert the dependencies. We might expect LoginController, LoginService and AnalyticsService to be declared in the same module controllers.login. And DefaultLoginService implementation class to be declared in a different one: services.login.

Now the module dependencies go in the reverse direction respect the function calls. The implementations depends on the abstract interfaces they implement. That's why this is called dependency inversion.
