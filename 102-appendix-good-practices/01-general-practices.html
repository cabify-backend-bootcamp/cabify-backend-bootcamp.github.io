General Good Practices

General Good Practices

In medicine, there are some good practices followed by doctors to heal their patients. Some of them are specific to the discipline. For example, in surgery they use to apply anaesthesia and monitor the patient. There are some other good practices that are non-specific to any discipline. Washing the doctor hands or work in a clean environment are some examples.

In software, there are also some general good practices that can be applied to the art of programming. We will discuss some of them along this section.

Don't Repeat Yourself (DRY)

Writing the same piece of code in several places is not a good idea. Let's see an example:

class AuthService {
  login(username, password) {
    let query = this.db.select("*").from("users").where("username == " + username);
    let rows = this.db.exec(query);
    if (rows.count == 0) {
      throw "user not exists";
    }
    // Do the login
  }

  changePassword(username, newPassword, oldPassword) {
    let query = this.db.select("*").from("users").where("username == " + username);
    let rows = this.db.exec(query);
    if (rows.count == 0) {
      throw "user not exists";
    }
    // Do the password change
  }
}

As we can see, our AuthService class can be used to login users or to change their passwords. But we want to check whether the user exists before proceeding. So we include a pre-condition check in both methods.

There are seveeral reasons that makes this particularly not recommended.

  • We are making the same work twice. In the best case, we are copying and pasting the lines. But that's also an extra work to be considered.
  • The code is more difficult to read. More lines, and particularly repeated, reduces the focus on the really important things.
  • This is a source of errors. It's fragile. Let's note that we might need to change how the user existence is checked in the code. So we should do it in both methods. We run the risk of changing the code in one place only, leaving the other one as it was. This is a really subtle bug that could be unnoticed for a long time.

So refactoring the code as follows is recommended.

class AuthService {
  login(username, password) {
    this.checkUserExists(username);
    // Do the login
  }

  changePassword(username, newPassword, oldPassword) {
    this.checkUserExists(username);
    // Do the password change
  }

  checkUserExists(username) {
    let query = this.db.select("*").from("users").where("username == " + username);
    let rows = this.db.exec(query);
    if (rows.count == 0) {
      throw "user not exists";
    }
  }
}

You Aren't Gonna Need It (YAGNI)

Code is not for free. It has a cost. A cost to write it, and a cost to maintain it. So the more code you have, the more you are investing on it. As most of us write code to support some business, we expect to have a as much return of investment as possible from it. This has a special implication: each block of code we write has to add value to the software. Otherwise, it should not be written.

Too often, we write code we think it's useful. Or it will be useful in the future. Typically adding functionality to the application that is not really needed. The return of investment of that is negative. We dedicate time to write and maintain that code when actually is not producing value to the business.

The solution: do not write functionality that is not needed. Implement the most functionaly simple thing that could work. But note: this is not a declaration to do the code as fast as possible, implement hacks and forget about clean code, tests, etc. This is about functionality. Do not implement functionality that was not required.

Keep It Simple Stupid! (KISS)

Every piece in a machine is a possible source of failure. Every gear, bearing, switch, button, bulb, can fail by its own. And when they fail, it's really likely that the rest of the machine turns inoperative. Also, many pieces working together require a bigger cognitive load. These systems are more difficult to understand, to learn how they work.

It is a maths rule. The more pieces you have in your machine, the more probability of failure. And the more difficulties to understand how the machine work. Software is not an exception. An application comprised by hundred of modules working together is more error prone and hard to understand than one made of a bunch of them.

That's why keeping things simple and avoiding unnecessary complexity is a good practice.

The Billion Dollar Mistake

In 1965, Tony Hoare was working on the design of a new programming language: ALGOL W. This was the first language to incorporate the concept of reference: a type system artifact that make a variable to "point" to a value. They designed the language so references could be null, so they point to no value. Since then, almost every programming language that support references accepts null (some call it nil instead).

In 2009, Tony Hoare apologised for inventing the null reference during a conference in London. He called that his Billion Dollar Mistake. Null references has led to innumerable errors, vulnerabilities, and system crashes, which have probably caused a billion dollars of pain and damage since invented.

What's wrong with null references? Well, the main problem is that very often we programmers use to make an assumption over references: they are not null. We define a class, define some methods...

class ScanService {
  // ...
  scan(product, amount) {
    // ...
    this.total += product.price * amount;
    // ...
  }
}

...and we assume product and amount are never null. But then, somehow, a null value is received in product. Perhaps because it was read from a JSON field that came through a HTTP request. And we didn't check it is null or not while unmarshalling (again: we don't expect that). So we end up executing the equivalent to this.total += null.price * amount. And that fails throwing:

TypeError: Cannot read property 'price' of null

This happens all the time. It's the top 1 of mistakes that make a system crash. Probably the impact in the software industry is much more than 1 billion dollars.

How can we deal with this? Well, it's not easy. Unfortunately, most programming languages consider that any variable can be null (everything is a reference, thus everything can be null). That's the case of Javascript, Ruby, Python, Java... . Nevertheless, there are some useful tips that help to mitigate the effects.

The first one is to put barries in your code to prevent the entrance of null values in your application. If you receive data from the network, from a file, from a database, make sure it contains no null values. Implement validations there and do not let any null to enter your domains, signalling the error (throw an exception, return HTTP 400 response, etc).

Once we have that, we will find we have to admit a variable can be null. For example, the email field of our User data class could be null to indicate we don't have the email for that user. In these cases, it could be convenient to use a placeholder constant that represents that null value. The placeholder is not a null reference, but a reference to an actual value that can behave like a regular one.

var NULL_EMAIL = "Email address was not set";

class User {
  constructor(firstName, lastName) {
    this.firstName = firstName;
    this.lastName = lastName;
    this.email = NULL_EMAIL;
  }
}

Thanks to that, we will have a defined behavior for any possible value of email field, even when its null. For instance, we can print the User properties without checking if things are null or not, so using our favorite HTML renderer:

<p>User information:</p>
<ul>
  <li>First name: {{ $user.firstName }}</li>
  <li>Last name: {{ $user.lastName }}</li>
  <li>Email: {{ $user.email }}</li>
</ul>

We can also compare email values to check if they are unset when needed. For instance, while sending emails:

function mailing_campaign(users, message) {
  users.forEach(function (user) {
    if (user.email != NULL_EMAIL) {
      send_email(user.email, message);
    }
  })
}

Another way to solve this is to use an option type. This is a data type present in some languages that can have one of these two states:

  • Some(x), where x is the value it wraps. It represents a non-null value.
  • None. It represents a null value.

Some examples are the Optional<T> generic class in Java 8 or the Option<T> enum type in Rust.

The point is that, when using statically typed languages (read this section for more details), we will interact with the interface of that optional type, forcing the programmer to consider that it can be none.

class User {
  private String firstName;
  private String lastName;
  private Optional<String> email;

  public User(String firstName, String lastName) {
    this.firstName = firstName;
    this.lastName = lastName;
    this.email = Optional.empty(); // This is None
  }

  public Optional<String> getEmail() { return this.email; }

  public void setEmail(String value) {
    this.email = Optional.of(value); // This is Some(value)
  }
  // ...
}

User user = new User("John", "Carpenter");
user.setEmail("john.carpenter@example.com");

Our email property is not String, but Optional<String>. So we have to use it accordingly:

void sendEmail(String address, String message) { ... }

void mailingCampaign(List<User> users, String message) {
  for (User user : users) {
    // ERROR: getEmail() is not String but Optional<String>
    // sendEmail(user.getEmail(), message)

    // OK: Optional implements ifPresent(), that will execute the action only when defined
    user.getEmail().ifPresent(email -> sendEmail(email, message));
  }
}
