Serving an API

Serving an API

Services are served by servers to clients

Computers talk to each other in all sorts of ways. For programming backends, we're not that interested in many of them; for example, we're not going to look into peer-to-peer communication models, such as BitTorrent or Bitcoin. (Although both are technically fascinating!)

The communication model that is most used between frontends and backends, and between different parts of backends, is the client-server model.

In the client-server model, there's some kind of service being, well, served. Then, there are two roles for participating computers:

  • Computers that use the service, called clients.
  • Those that serve the service, called servers.

We'll explore the relationship between clients and servers in detail in this chapter.

Protocols: the language in which computers talk

Clients and servers must share a common understanding on how to communicate: a protocol.

In practice, a protocol is really a composition of protocols from different layers. On low-level layers, computers agree on the basics about how to communicate. By using standard protocols for those layers, you can get advantage of existing tools that know about those protocols, such as routers, operating systems and web browser, and focus on what makes your use case different.

There are protocols that:

  • Work out the details of how to find a computer in the network: IP, DNS.
  • Just move data between computers: TCP (reliable, but heavyweight), UDP (not reliable, but lightweight).
  • Encrypt the data, to avoid tampering and ensure privacy: TLS.
  • Describe ways of shaping this data into messages, and how messages are sequenced: HTTP, JSON-RPC, gRPC, WebSockets.

Ways of serving

Different protocols implement different patterns of communication between clients and servers. Let's explore some of those.

Requests get responses

The most common way of serving services is by accepting messages from clients, called requests, that specify what they want the service to do for them; and, after some work is done, sending a message back to the client, called a response, with some kind of feedback about the request.

sequenceDiagram client ->>+ server: make me a sandwich activate client note right of server: *mumble* *rumble* server ->>- client: 🥪 deactivate client

This is, for instance, how HTTP works. Each time you visit a web page, you're being a client of a service; the browser sends a HTTP request to the servers, that respond with content.

Don't push to me, I'll pull from you

When you go to the hospital, you don't just immediately go into the doctor's office, sit down and start telling them your pains and sorrows right away, without even minding if there's already some other patient in there. Instead, you wait in a room for the doctor to call you when they're ready to serve your request.

This is pull-based serving, as opposed to push-based. It's mostly used when, rather than expecting a meaningful response from the server, the client provides the data to the server, without being really interested in a response from it.

Here, the request is sent from the server to the client, and the client responds to it.

sequenceDiagram participant client participant server server ->> client: need your data client ->> server: have all this data, enjoy

The advantage of pulling data instead of waiting for it to be pushed is that the server decides when to pull it, which can be more efficient, since the server knows better than the client when it's preferable to have the data.

Streams of responses

Sometimes, instead of expecting a response to a request as soon as possible, clients expect a series of responses, being sent whenever they are ready. Think something like chat: as people post messages, they're delivered to you.

sequenceDiagram client ->>+ server: connect to chat activate client server ->> client: OK, hang on for messages note right of server: (Some time passes.) Alice ->>+ server: hi! activate Alice server ->>- Alice: OK deactivate Alice server ->> client: Alice says: hi! note right of server: (Some time passes.) Bob ->>+ server: I'm here too! activate Bob Alice ->>+ server: how are you? activate Alice server ->>- Alice: OK deactivate Alice server ->> client: Alice says: how are you? server ->>- Bob: OK deactivate Bob server ->> client: Bob says: I'm here too! deactivate client

This is useful when the client wants to be notified when events happen as soon as possible after they happen.

A downside is that both client and server need to keep the stream open, which, as we'll extensively see, consumes system resources.

Web protocols that enable streams: SSE, WebSocket.

Long polling

You can trick a request-response protocol into behaving like a stream. The trick is to send a request and, instead of expecting a response right away, let the server respond only once a new event arrives. Once you get the response, you send the request again.

sequenceDiagram client ->>+ server: get me messages activate client note right of server: (Some time passes.) Alice ->>+ server: hi! activate Alice server ->>- Alice: OK deactivate Alice server ->>- client: Alice says: hi! deactivate client client ->>+ server: get me messages activate client note right of server: (Some time passes.) Bob ->>+ server: I'm here too! activate Bob Alice ->>+ server: how are you? activate Alice server ->>- Alice: OK deactivate Alice server ->>- client: Alice says: how are you? deactivate client server ->>- Bob: OK deactivate Bob client ->>+ server: get me messages activate client server ->>- client: Bob says: I'm here too! deactivate client

The main advantage of this is that you can use a request-response protocol, such as HTTP, which are more ubiquitous and thus are familiar to more clients developers and have more libraries available for all kinds of platforms.

Stream from A to B + stream from B to A = connection

If, instead of from server to client only, messages are streamed in both directions, we have a connection.

Going back to the chat application, we had:

  • A stream-based protocol for retrieving messages.
  • A request-response protocol for posting messages.

It's more efficient to just have a single connection for both. This has basically no resources overhead over plain streams, and spares us the overhead of the request-response protocol.

A connection-based protocol available in Web browsers is WebSockets.

Application Programing Interface

Once you've decided how your server will communicate with its clients, you must choose a stack of standard protocols that best match your needs, and, on top of those, add a protocol that you define. The protocol that is specific to an application is usually called the API.

The API is the external interface that a system exposes to let others build programs on top of it.

It is different from a user interface, which is supposed to be used by a person directly through human-friendly input and output methods: buttons, commands, text, images, etc.

Instead, an API is supposed to be used programmatically, by writing code.

In the case of servers, APIs are exposed to clients through the network, on top of other protocols that deal with communication details.

Even though APIs are for machines to call, it's people the ones writing the code that deals with them. Bad APIs make people miserable and systems brittle. While what constitutes a good API is, as with most aspects of human interaction, fairly subjective, there are a few principles that are a necessary factor in designing good APIs.

An API is a contract

The first and most crucial requirement for a good API is that it is well defined.

APIs get old

Here's a fact of life: you will want to change your API. Here's another: you won't be able to, because there will be clients depending on it.

The code your clients are using to access your old API won't magically change at exactly the same time your server stop serving it. Mobile apps won't get updated, browser tabs won't refresh, services depending on yours won't be deployed until after some time passes, or, in the worst case, never.

Backwards-compatible additions

Instead of changing your API, consider adding things to it.

State very clearly which kind of additions are considered backwards-compatible. A good list would include:

  • Adding kinds of requests: methods, endpoints, resources, etc.
  • Adding fields to object types.
  • Adding parameters to kinds of requests, if the protocol allows it: query parameters in URLs, optional arguments in methods, etc.

Versioning

When breaking changes can't be avoided, it's best to keep running the old API alongside the new one.

Clients should be told that the should update to take advantage of the new, better API.

Programmer-friendliness

In the end, a programmer is the one that needs to understand and adapt to your API. A good API is designed with this in mind.

Being friendly to people is more art than science. It entails things like being simple, powerful, easy to use, flexible, etc. But let's see a few recommendations that are usually programmer-friendly APIs.

Follow conventions

Yours is not the only API programmers have seen or will see in their lives. If your API resembles those they already knew, you can leverage that familiarity in order to reduce friction.

There will probably be libraries, tools, etc. built around those conventions.

Give good error feedback

Use precisely codified errors and informative error messages in your responses to faulty requests.

Keep ceremony to a minimum

Set sane defaults when they make sense. But careful: this is a slippery slope. You might end up doing things by default that are annoying or surprising to many people.

Provide client libraries

Specially when your API is complex, providing a client library may be a good idea.

A client library should not be substitute for a good platform-agnostic API: platforms change, and you might end up inducing more friction.

TCP connections and your API

Ideally, when serving your API, you wouldn't need to care too much about the underlying protocols. Things would just work beneath you.

In practice, when things go wrong, those foundational protocols break and your API is affected. If you want your API to be well-defined, you can't escape from that fact. Effectively, the underlying protocols are part of your API.

In particular, TCP is such a fundamental protocol that is is worth knowing in some detail how it affects your API.

Delivery guarantees

As long as the connection is not broken, TCP ensures that the data you send is received:

  • Eventually
  • Only once
  • In the order it was sent
  • Incorrupt

Server: cancel work when the connection dies

There are three ways a connection should be declared dead:

  • You receive a FIN when you expected the other end to keep sending data. In Node.js, this manifests itself as an end event emitted on the net.Socket or a http.Client object.
  • You receive a RST. This means the other end has declared the connection as broken and tells you to do so too, for whatever reason. This is the "connection reset by peer" error that manifests itself as an error event emitted on the request object.
  • The connection times out.

You should listen to those events, propagate them, and stop doing things in your request handler as soon as possible.

Client: close the connection if you don't need a response anymore

In the browser's XMLHTTPRequest, call the abort method.

In Node.js's http.ClientRequest, call the destroy method.

RPC

RPC means Remote Procedure Call.

It basically means that you send a request and get a response, similarly to how you would call a procedure (a method or a function) and get a return value in your programming language.

HTTP as a transport

HTTP is a protocol on top of TCP designed to interact with hypertext, that is, Web pages. But, since there's a lot of tooling already in place for HTTP, it's usual to rely on it as a transport layer for RPC.

JSON-RPC

JSON-RPC is a particular way of encoding RPC calls in HTTP, using JSON.

The nice thing about JSON-RPC is that there are tons of libraries available for it, in many platforms.

REST

In REST, you define your API in terms of resources. A resource is something that can be retrieved, maybe modified, and maybe deleted.

CRUD operations:

  • GET
    • Reads a resource.
    • Cacheable.
    • Must be idempotent.
  • POST
    • Writes a resource, or provoke an effect of some kind.
    • Not necessarily idempotent.
      • But you need to make it idempotent if you want strong delivery guarantees, to safely support retries.
  • PUT
    • Writes a resource by replacing it wholly.
    • Must be idempotent.
  • DELETE
    • Deletes a resource.
    • Must be idempotent.
  • PATCH
    • Replaces just part of a resource. Pretty error-prone; usually not worth it.

This is how browsers work, mostly.

Virtually all platforms have good support for RESTful APIs.

Provides familiar semantics. Or that's the myth. In practice, it's not that well understood, and you'll find yourself shoehorning things into it, since not everything can be expressed in terms of resources.

Because of this, just declaring your API as "RESTful" doesn't mean there's no need to define your API in very precise terms.

REST also requires you to respond with semantically valid status codes:

  • 200-299: Things went well.
  • 400-499: Something went bad, because of the client.
  • 500-599: Something went bad, because of the server.

You can get fancy, or you can stick to:

  • 200 OK
  • 400 Bad Request
  • 401 Unauthorized
  • 403 Forbidden
  • 404 Not Found
  • 500 Internal Server Error
  • 503 Service Unavailable

Deriving code from contracts

Many tools let you derive both client and server code from a common language that specifies part of your contract. This is nice because it enforces that part of the contract in a uniform, robust way, so that you can focus on the other parts.
